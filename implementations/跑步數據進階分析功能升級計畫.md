# 跑步數據進階分析功能升級計畫

## 目標
解決目前「輕鬆跑、間歇跑、乳酸閾值跑」只看總平均數據（距離、時間、心率、配速）導致的分析失真問題。讓 Gemini 教練能夠根據真實的訓練強度與型態給出精準的建議。

---

## 背景與痛點
間歇跑或乳酸閾值跑因為包含熱身 (Warm-up) 與緩和 (Cool-down) 階段，這會大幅拉低整體的「平均心率」與「平均配速」。如果教練只看平均值，可能會誤判「你這次跑得太輕鬆」，而忽略了主課表的高強度區間。

## 解決方案：引入 Garmin 的進階數據
根據 Garmin API 提供的原始資料，我們其實有非常豐富的數據可以運用。我提議將 `get_recent_runs.py` 獲取的資料擴充，並一併應用在 `main.py` 的跑後自動分析推播中。

### 新增抓取的關鍵指標：

1. **訓練效果指標 (Training Effect)**：
   - `trainingEffectLabel`：Garmin 直接幫你分類這趟跑是什麼類型（例如：`RECOVERY`, `BASE`, `TEMPO`, `THRESHOLD`, `VO2MAX`, `ANAEROBIC`）。
   - `aerobicTrainingEffect` / `anaerobicTrainingEffect`：有氧與無氧的訓練壓力分數 (0.0 ~ 5.0)。
   *(這能讓教練一眼看出這趟是不是間歇跑，以及有沒有達到無氧刺激)*

2. **極限與強度指標 (Intensity)**：
   - `maxHR`：最高心率。
   - `maxSpeed`：最高配速（衝刺時的最快速度）。
   *(這能證明在間歇段落你有推到極限，打破「平均心率低」的假象)*

3. **跑步動態數據 (Running Dynamics)**：
   - `averageRunningCadenceInStepsPerMinute`：平均步頻（步/分鐘）。
   - `avgStrideLength`：平均步幅（公分）。
   *(步頻與步幅是判斷跑步技術、受傷風險的核心指標)*

## 具體修改項目 (Proposed Changes)

### 1. `scripts/get_recent_runs.py` [MODIFY]
修改終端機輸出的格式，提供給 Gemini CLI 更多維度的資料：
```text
📅 2026-05-03 | 🏃 桃園區 跑步
   [基礎數據] 距離: 2.18 km | 時間: 16.5 分 | 平均配速: 7:35 /km
   [訓練成效] 標籤: RECOVERY | 有氧 TE: 1.8 | 無氧 TE: 0.4
   [強度指標] 平均心率: 119 bpm | 最高心率: 162 bpm | 最快配速: 4:08 /km
   [跑步動態] 平均步頻: 153 spm | 步幅: 86 cm
```

### 2. `main.py` (跑後自動分析推播) [MODIFY]
在 `_post_run_polling` 函式中，把提示詞 (Prompt) 加上上述的進階數據。
讓系統發給 Gemini 的內容變成：
> "我剛完成一次跑步，Garmin 判定這是一次 **[VO2MAX]** 訓練。有氧效果 3.5，無氧效果 2.1。平均心率 145，但最高心率達到 185。平均步頻 175... 請根據這些數據給我教練建議。"

### 3. `SKILL.md` 教練指南更新 [MODIFY]
在教練的 `SKILL.md` 中加入解讀 `Training Effect Label` 和 `最高心率` 的邏輯，教導 LLM「不要只看間歇跑的平均配速」。

## User Review Required

> [!IMPORTANT]
> **分段資料 (Lap / Splits) 是否需要納入？**
>
> 目前的升級計畫利用「最高心率」和「無氧效果 TE」已經能很好地解決間歇跑被誤判的問題。
> 另一個更進階的做法是把每一圈 (Lap) 的資料都抓出來（例如 400m 衝刺 x 8 趟的每趟配速），但這會讓資料量變得非常大，且增加 API 的複雜度。
>
> **建議：** 先實作上述的「訓練成效標籤 (TE Label) + 最高強度 + 跑步動態」版本，這已經能讓教練變聰明非常多。如果未來覺得不夠，我們再來考慮撈取 Lap 分段資料。您同意這個做法嗎？
